# Consumer Contract

## Overview

Consumer Contract is the documentation surface for how Cedar token outputs are consumed.
Each output type has its own contract page describing what it is, why Cedar publishes it,
when a consumer should choose it, and how to import it.

### Contract outputs

- type
- css
- scss
- js
- json

### Consumer goal

Pick the narrowest output contract that fits your integration and avoid coupling to internal build details.

### Recommended default

Use the js semantic interface (`@rei/cdr-tokens`) for long-lived app-level token mappings.

## Usage Examples

### Example 1: App theme mapping (js + semantic)

Map semantic contracts into your app token layer.

```ts
import { CdrColorText, CdrSpace, CdrType } from '@rei/cdr-tokens';

export const appTheme = {
  color: {
    textPrimary: CdrColorText.CdrColorTextPrimary,
  },
  space: {
    stackMd: CdrSpace.CdrSpaceTwoX,
  },
  type: {
    headingMd: CdrType.CdrTextHeadingSans400FontSize,
  },
};
```

### Example 2: Map typography into an app contract

Use composite typography objects when the destination API expects complete text recipes.

```ts
import { CdrText, CdrType } from '@rei/cdr-tokens';

export const textStyles = {
  body: {
    fontSize: CdrText.CdrTextBody300FontSize,
    fontWeight: CdrText.CdrTextBody300FontWeight,
    lineHeight: CdrText.CdrTextBody300LineHeight,
  },
  heading: {
    fontSize: CdrType.CdrTextHeadingSans400FontSize,
    fontWeight: CdrType.CdrTextHeadingSans400FontWeight,
    lineHeight: CdrType.CdrTextHeadingSans400LineHeight,
  },
};
```

### Example 3: Stylesheet integration (css/scss)

Use CSS for simple imports and SCSS when mixins/utilities are needed.

```ts
import '@rei/cdr-tokens/css';
```

```scss
@use '@rei/cdr-tokens/scss';
@use '@rei/cdr-tokens/scss/breakpoint-mixins' as bp;
```

### Example 4: Pipeline ingestion (json)

Read raw artifacts for tooling, transforms, or audits.

```ts
import globalTokens from '@rei/cdr-tokens/json/global.json';
import webTokens from '@rei/cdr-tokens/json';

console.log({ globalTokens, webTokens });
```

## Consumer Contracts

## type

### what it is

Type contracts for generated token modules with inline IDE metadata.

Cedar publishes `.d.ts` type definition files alongside every runtime `.mjs` module. Two things make them useful beyond basic type safety:

1. **Every property carries `@value`, `@cssvar`, and a description** — generated by the Cedar build from token source. Your IDE reads these and shows them in hover tooltips and autocomplete without any extension or plugin.
2. **The semantic root (`@rei/cdr-tokens`) surfaces grouped object types** — hovering `CdrColorText.` in VS Code shows the full `CdrColorTextTokens` interface with all members documented.

There is not currently a published `index.d.ts` entrypoint for consumers to import directly.
Instead, the package exposes types through three real entrypoints:

- `@rei/cdr-tokens` → typed by `dist/rei-dot-com/types/index.d.ts` (semantic default)
- `@rei/cdr-tokens/tokens` → typed by `dist/rei-dot-com/js/cdr-tokens.d.ts` (full flat surface)
- `@rei/cdr-tokens/types/*` → typed by `dist/rei-dot-com/types/foundations/cdr-*.d.ts`

### why we provide it

The primary goal is developer experience at the point of use — a consumer writing `CdrColorText.` should be able to choose the right token without leaving their editor or opening the Cedar docs site.

Cedar achieves this by embedding `@value`, `@cssvar`, and description text directly in the generated `.d.ts` files. The TypeScript Language Server renders these as hover tooltips in any LSP-supporting editor (VS Code, WebStorm, Neovim+LSP). No extension or configuration is required.

Cedar also owns the output shape, so consumers do not need custom interfaces or any-casts.

### IDE hover example

The generated `.d.ts` format that your IDE reads:

```ts
export interface CdrColorTextTokens {
  /**
   * The default, primary text color
   * @value #4b4a48
   * @cssvar --cdr-color-text-primary
   */
  readonly CdrColorTextPrimary: string;
}
```

What VS Code shows when you hover `CdrColorText.CdrColorTextPrimary`:

```
(property) CdrColorTextPrimary: string

The default, primary text color
@value — #4b4a48
@cssvar — --cdr-color-text-primary
```

### when to use it

Use this when your consumer code is TypeScript and you want typed token surfaces in theme files, design token adapters, or utility layers.

Use the `.mjs` module when you need the actual token values at runtime.
Use the `.d.ts` file when you want to explicitly annotate the shape of a token object in TypeScript.

In most app code, you import the value from `@rei/cdr-tokens` and let TypeScript infer the types automatically. Reach for the `.d.ts` path only when you need an explicit contract type.

The semantic root `@rei/cdr-tokens` gives the best IDE experience for token discovery. Use `@rei/cdr-tokens/tokens` for flat lookups and `@rei/cdr-tokens/types/*` for narrow domain imports.

### how to use it

Use the entrypoint that matches your integration intent:

- `@rei/cdr-tokens`: semantic default for app theme contracts (recommended)
- `@rei/cdr-tokens/tokens`: full flat output when you need every token directly
- `@rei/cdr-tokens/types/*`: focused domain module when you want a narrow import boundary

### Example A: semantic root (default)

Use this for most app integrations. It is the easiest surface for autocomplete and long-lived semantic mappings.

```ts
import { CdrColorText, CdrSpace, CdrType } from '@rei/cdr-tokens';

export const appTheme = {
  textPrimary: CdrColorText.CdrColorTextPrimary,
  spacingMd: CdrSpace.CdrSpaceTwoX,
  headingMd: CdrType.CdrTextHeadingSans400FontSize,
};
```

### Example B: semantic root with explicit type annotation

Use this when you want compile-time contract enforcement in shared theme interfaces.

```ts
import type { CdrColorTextTokens } from '@rei/cdr-tokens/types/color-text';
import { CdrColorText } from '@rei/cdr-tokens';

const text: CdrColorTextTokens = CdrColorText;
```

### Example C: full flat surface via `/tokens`

Use this for one-off lookups, audits, scripts, or tooling that needs the complete token surface.

```ts
import * as tokens from '@rei/cdr-tokens/tokens';

const primary = tokens.CdrColorTextPrimary;
const buttonRest = tokens.CdrColorBackgroundButtonPrimaryRest;
```

### Example D: focused domain module via `/types/*`

Use this when you want to limit imports to a specific foundation domain.

```ts
import { CdrColorText } from '@rei/cdr-tokens/types/color-text';

const textPrimary = CdrColorText.CdrColorTextPrimary;
```

### `.mjs` vs `.d.ts`

| File type | What it contains            | Use it when                                          | Do not use it when                             |
| --------- | --------------------------- | ---------------------------------------------------- | ---------------------------------------------- |
| `.mjs`    | Runtime token values        | You need actual values in app code                   | You only need a type annotation                |
| `.d.ts`   | TypeScript type definitions | You want to annotate a variable/interface explicitly | You are trying to read token values at runtime |

### Published type entrypoints

| Import path               | Published type file                             | Best use                                                         |
| ------------------------- | ----------------------------------------------- | ---------------------------------------------------------------- |
| `@rei/cdr-tokens`         | `dist/rei-dot-com/types/index.d.ts`             | Semantic default app contract and IDE-friendly grouped discovery |
| `@rei/cdr-tokens/tokens`  | `dist/rei-dot-com/js/cdr-tokens.d.ts`           | Full flat token surface for scripts/audits/one-off lookups       |
| `@rei/cdr-tokens/types/*` | `dist/rei-dot-com/types/foundations/cdr-*.d.ts` | Narrow domain boundary with explicit foundation-level typing     |

### Example: semantic + flat together

```ts
import type { CdrColorTextTokens } from '@rei/cdr-tokens/types/color-text';
import * as flatTokens from '@rei/cdr-tokens/tokens';
import { CdrColorText } from '@rei/cdr-tokens';

export interface ThemeContract {
  color: CdrColorTextTokens;
}

export const theme: ThemeContract = {
  color: CdrColorText,
};

export const diagnostics = {
  primaryText: flatTokens.CdrColorTextPrimary,
};
```

## css

### what it is

Prebuilt CSS bundles and foundation-level CSS modules published under the dist output.

### why we provide it

This gives consumers a no-transform path to Cedar tokens and avoids building custom CSS token pipelines for simple integrations.

### when to use it

Use this when the app consumes plain CSS, uses bundler CSS imports, or needs a specific foundation module without Sass.

### how to use it

```ts
import '@rei/cdr-tokens/css';
import '@rei/cdr-tokens/css/color-text';
```

## scss

### what it is

Sass token outputs plus Cedar utility mixins/maps under the dist SCSS output.

### why we provide it

It supports Sass-native workflows where consumers need mixins and utility contracts directly in stylesheets.

### when to use it

Use this when your application compiles Sass and you want Cedar utilities for type, spacing, and breakpoints.

### how to use it

```scss
@use '@rei/cdr-tokens/scss';
@use '@rei/cdr-tokens/scss/type-mixins' as typeMixins;
@use '@rei/cdr-tokens/scss/breakpoint-mixins' as breakpoints;
```

## js

### what it is

JavaScript value contracts available at the root entrypoint and semantic entrypoint.

### why we provide it

This is the primary runtime integration surface for framework mapping, app theming objects, and token-driven UI config.

### when to use it

Use this when you are mapping tokens in JavaScript/TypeScript application code and want either full output access or stable semantic access.

### how to use it

```ts
import * as tokens from '@rei/cdr-tokens/tokens';
import { CdrColorText, CdrSpace, CdrType } from '@rei/cdr-tokens';

tokens.CdrColorTextPrimary;
CdrColorText.CdrColorTextPrimary;
CdrSpace.CdrSpaceOneX;
CdrType.CdrTextHeadingSans400FontSize;
```

## json

### what it is

Raw JSON token artifacts generated by theme/platform in the dist JSON output.

### why we provide it

These files are useful for tooling, audits, custom transforms, and consumers outside a JS module import workflow.

### when to use it

Use this for pipeline automation, design token transforms, and environments that expect JSON sources.

### how to use it

```ts
import globalTokens from '@rei/cdr-tokens/json/global.json';
import webTokens from '@rei/cdr-tokens/json';

console.log(globalTokens, webTokens);
```
