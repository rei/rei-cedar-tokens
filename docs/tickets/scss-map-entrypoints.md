# Ticket: SCSS Dual Map Entrypoints (Q2)

## Summary

Split `utility-map.scss` into two parallel map families — resolved-value maps and `var()` reference maps — and expose them as separate, named package export entrypoints.

## Background

Decision recorded in `docs/tickets/semantic-contract-consumer-questions.md` → Q2.

`utility-map.scss` currently emits resolved rem/px values:

```scss
$breakpoints: (
  xs: 0,
  sm: 768px,
  md: 992px,
  lg: 1232px,
);
$radius: (
  sharp: 0,
  soft: 4px,
  ...,
);
```

Consumers that need `calc()` arithmetic require resolved values. Consumers that need runtime theme switching (e.g. dark mode via CSS custom property overrides) require `var(--cdr-*)` values. These two shapes are structurally incompatible in a single map — the same SCSS variable cannot safely serve both.

## Acceptance Criteria

1. `dist/rei-dot-com/scss/map-resolved.scss` emits all utility maps with resolved values (current behavior — extracted from `utility-map.scss`).
2. `dist/rei-dot-com/scss/map-vars.scss` emits the same map keys but with `var(--cdr-*)` values.
3. Both files are generated by `style-dictionary/build.ts` — not handwritten.
4. Map key names are identical across both files so consumers can switch imports with no further code changes.
5. `package.json` exports includes two new entrypoints:
   ```json
   "./scss/map-resolved": "./dist/rei-dot-com/scss/map-resolved.scss",
   "./scss/map-vars":     "./dist/rei-dot-com/scss/map-vars.scss"
   ```
6. `utility-map.scss` is either deprecated (emitting `@warn`) or removed and replaced by the two new files. If deprecated, note the replacement in the warning.
7. At minimum, the following map families must be present in both outputs: `$breakpoints`, `$radius`. Extend to `$spacing`, `$color-background`, `$color-border`, `$color-text` as those foundation modules are consumed.
8. CI `pnpm build` produces both files without error.
9. A brief entry is added to `docs/TRANSFORMS.md` documenting the two entrypoints and when to use each.

## Implementation Notes

- The `var()` map values should mirror the CSS custom property names already emitted in `dist/rei-dot-com/css/foundations/cdr-*.css`. Derive CSS var names from token names — do not hardcode them.
- Resolved values already exist in `utility-map.scss`. The generator logic for `map-resolved.scss` should be a refactor of that existing logic, not a rewrite.
- Use Style Dictionary's `value` field (resolved) for `map-resolved`, and construct `var(--cdr-${token.name})` strings for `map-vars`.
- Scope initial implementation to `rei-dot-com` surface only. Docsite surface can follow.

## Scope

- `style-dictionary/build.ts` — add `map-resolved` and `map-vars` format/action
- `dist/rei-dot-com/scss/map-resolved.scss` — generated artifact
- `dist/rei-dot-com/scss/map-vars.scss` — generated artifact
- `package.json` — two new export entries
- `docs/TRANSFORMS.md` — usage note
- `utility-map.scss` — deprecation `@warn` or removal
